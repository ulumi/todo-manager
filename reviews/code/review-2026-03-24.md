## 1. Sécurité

### 🔴 CRITIQUE — Clé de service Firebase exposée dans le dépôt git

**Fichier :** `firebase-service-account.json`

La clé privée du compte de service Firebase (`private_key`, `private_key_id`, `client_email`) est présente en clair dans le dépôt. Ce fichier donne un accès administrateur complet à Firestore, Firebase Auth, et tous les services GCP associés au projet `huge-todo`.

**État actuel :** Le fichier est dans `.gitignore`, donc il n'est **pas committé** — mais il est présent sur le disque et risque d'être accidentellement versionné si `.gitignore` est modifié ou si quelqu'un utilise `git add -f`.

**Risque :** Si ce fichier est jamais poussé sur GitHub ou un remote partagé, n'importe qui peut lire/écrire tous les documents Firestore, créer/supprimer des comptes utilisateurs, et accéder aux données privées de tous les utilisateurs.

**Recommandation :**
- Vérifier que le fichier n'a jamais été commis (`git log --all -- firebase-service-account.json`).
- En production, la clé doit transiter uniquement via la variable d'environnement `FIREBASE_SERVICE_ACCOUNT` (Vercel), jamais en fichier local versionnable.
- Envisager une rotation de la clé par précaution.

### 🔴 CRITIQUE — CORS trop permissif sur l'API admin

**Fichier :** `api/admin-users.js` (ligne 18), tous les fichiers `api/admin-*.js`

```js
res.setHeader('Access-Control-Allow-Origin', '*');
```

Les endpoints admin (`/api/admin-users`, `/api/admin-batch`, `/api/admin-user-action`, etc.) acceptent des requêtes de n'importe quelle origine. Même si l'authentification par token Firebase protège les endpoints, une origine restreinte au domaine de production (`https://todo.hugues.app`) serait une défense en profondeur correcte.

**Recommandation :** Restreindre l'en-tête à l'origine de l'app en production, ou au moins à un ensemble connu d'origines.

### 🟡 Injection potentielle dans les attributs `onclick` via les noms de catégories/projets

**Fichier :** `js/modules/render.js` (lignes 36, 42, 1097, 1166, 1403, 1560)

Les `cat.id` et `proj.id` sont des timestamps numériques (`Date.now().toString()`), donc sûrs. Mais les `cat.color` sont copiés directement dans des attributs `style=""` sans échappement :

```js
return `<span ... style="background:${cat.color};...">`;
```

Si un utilisateur malveillant injectait une valeur `color` corrompue dans Firestore (accès direct à la DB, ou bug dans une autre route), cela pourrait injecter du CSS arbitraire. Le vecteur est limité (l'utilisateur attaquerait ses propres données), mais mérite d'être noté.

---

### 🟡 Escape incomplet dans `_renderQuickFindDropdown` — XSS potentiel

**Fichier :** `js/app.js` (ligne 477)

```js
const qEsc = query.replace(/'/g, "\\'");
// ...
onclick="window.app.openSearchView('${qEsc}')"
```

Seuls les apostrophes sont échappés. Un `query` contenant `"` (guillemet double) peut sortir de l'attribut HTML `onclick="..."` et injecter du HTML arbitraire. L'échappement correct serait `esc(qEsc)` (la fonction `esc()` du projet qui encode `&`, `<`, `>`, `"`).

---

### 🟡 Contenu non échappé dans le panel debug Celebrate

**Fichier :** `js/app.js` (ligne 1112)

```js
panel.innerHTML = `
  ...
  <div ... id="debugQuote" contenteditable="false">${quote}</div>
  ...
  onclick="window.app._banMascot('${mascot}')">Ban</button>
```

La `quote` et le `mascot` sont insérés en HTML brut. Les quotes proviennent de la base Firestore (via `getGlobalQuotes`). Ce panel est réservé aux admins, mais c'est quand même une XSS stored si un admin malveillant ou un attaquant avec accès Firestore peut injecter du HTML dans les quotes.

## 2. Bugs

### 🔴 Bug logique — `toggleInboxDone` : celebrate déclenché à l'envers

**Fichier :** `js/app.js` (ligne 1266)

```js
toggleInboxDone(id) {
  // ...
  t.completed = !t.completed;
  saveTodos(state.todos);
  if (!t.completed) celebrate(state.lang);  // ← BUG
  this.render();
}
```

La mutation `t.completed = !t.completed` est faite **avant** le test. Donc après la ligne de mutation, si `t.completed` est `true`, c'est qu'on vient de **compléter** la tâche (avant elle était `false`). Le celebrate devrait se déclencher quand `t.completed === true`, pas `false`.

Comparaison avec `toggleTodo` (ligne 938) qui fait correctement :
```js
const wasCompleted = isCompleted(...);
// ...
if (!wasCompleted) celebrate(state.lang);
```

**Fix :** Capturer l'état avant la mutation, ou inverser la condition :
```js
if (t.completed) celebrate(state.lang);  // celebrate si on vient de compléter
```

---

### 🔴 Bug — `navigate()` en vue "week" saute 14 jours au lieu de 7

**Fichier :** `js/app.js` (ligne 772)

```js
if (state.view==='week')  d.setDate(d.getDate()+delta*14);
```

La vue "week" dans `renderWeekView` affiche **2 semaines** (week1 + week2 partant de `startOfWeek(navDate)`). Naviguer de 14 jours semble intentionnel pour avancer d'un bloc de 2 semaines. Cela dit, cela signifie qu'on ne peut jamais voir la semaine intermédiaire directement — si l'utilisateur est en semaine 1-2 et navigue, il passe à la semaine 3-4. C'est cohérent avec le comportement voulu mais mérite d'être documenté car c'est surprenant.

**Si l'intention est de naviguer semaine par semaine :** remplacer `*14` par `*7`.
**Si c'est intentionnel (bi-semaine) :** ajouter un commentaire explicatif.

---

### 🟡 Bug potentiel — `deleteFutureOccurrences` ne retourne pas `todos` dans le cas sans endDate

**Fichier :** `js/modules/calendar.js` (lignes 60-70)

```js
export function deleteFutureOccurrences(id, date, todos) {
  const t = todos.find(x => x.id === id);
  if (t) {
    const endDate = DS(addDays(date, -1));
    if (t.startDate && endDate < t.startDate) {
      return todos.filter(x => x.id !== id);  // retourne
    } else {
      t.endDate = endDate;  // mutation en place, PAS de return explicite
    }
  }
  return todos;  // retourne tout de même
}
```

Ce n'est pas un bug en soi (la mutation en place + `return todos` fonctionne), mais le pattern est incohérent : un chemin retourne un nouveau tableau filtré, l'autre mute en place et retourne le tableau original. L'appelant (`app.js` ligne 984) fait :

```js
const newTodos = deleteFutureOccurrences(id, state.pendingDelete.date, state.todos);
state.setTodos(newTodos);
```

Cela fonctionne uniquement parce que dans le cas "mutation en place", `newTodos === state.todos`. C'est fragile.

## 3. Persistance & Synchronisation (Firestore)

### 🔴 `saveCategories()` dans `admin.js` n'appelle pas `pushFirestoreNow()`

**Fichier :** `js/modules/admin.js` (ligne 41-43)

```js
export function saveCategories(categories) {
  localStorage.setItem(PROJECTS_KEY, JSON.stringify(categories));
  // ← PAS de pushFirestoreNow() !
}
```

Toutes les opérations qui passent par `saveCategories()` directement (et non via `app.js`) ne synchronisent pas Firestore :
- `addCategory()` → `saveCategories()` (admin.js ligne 52)
- `removeCategory()` → `saveCategories()` (admin.js ligne 59)
- `addCategoryInline()` → `saveCategories()` (modal.js ligne 201)

Par comparaison, `saveProjects()` dans `projectManager.js` appelle bien `pushFirestoreNow()` (ligne 36).

Les appels passant par `app.js` (`setCategoryColor`, `saveCategoryName`, `addCategoryFromView`) n'appellent pas non plus `pushFirestoreNow()` explicitement après `saveCategories()`.

**Impact :** Créer ou supprimer une catégorie ne se synchronise pas vers Firestore. Si l'utilisateur change d'appareil ou vide son localStorage, les catégories sont perdues.

**Fix :** Ajouter `pushFirestoreNow()` dans `saveCategories()` (avec l'import correspondant depuis `storage.js`).

---

### 🔴 `saveTemplates()` dans `admin.js` n'appelle pas `pushFirestoreNow()`

**Fichier :** `js/modules/admin.js` (ligne 93-95)

```js
export function saveTemplates(templates) {
  localStorage.setItem(TEMPLATES_KEY, JSON.stringify(templates));
  // ← PAS de pushFirestoreNow() !
}
```

Toutes les modifications de templates (ajout, suppression, ajout de tâche, retrait de tâche) ne se synchronisent pas vers Firestore.

### 🟡 `saveSuggestedTasks()` dans `admin.js` n'appelle pas `pushFirestoreNow()`

**Fichier :** `js/modules/admin.js` (ligne 214-216)

Même problème pour les tâches suggérées.

La synchronisation **inverse** (Firestore → local) pour les catégories fonctionne bien via `_applyBackup` (ligne 4497). Le problème est uniquement dans le sens local → Firestore.

## 4. Performance

### 🟡 Re-render complet à chaque changement de localStorage (cross-tab sync)

**Fichier :** `js/app.js` (lignes 175-188)

```js
case 'projects':
case 'dayTemplates':
case 'suggestedTasks':
case 'projectTaskOrder':
case 'categoriesCols':
case 'categoriesSort':
  this.render();
  break;
```

`this.render()` est un re-render complet du DOM. Pour des changements comme `categoriesCols` ou `categoriesSort`, reconstruire tout le DOM (y compris re-attacher tous les drag-and-drop, etc.) est coûteux. Dans la pratique, ces changements sont rares, donc l'impact est faible. À surveiller si le projet grossit.

---

### 🟡 `getCategories()` re-parse le JSON à chaque appel

**Fichier :** `js/modules/admin.js` (lignes 36-39)

```js
export function getCategories() {
  const stored = localStorage.getItem(PROJECTS_KEY);
  return stored ? JSON.parse(stored) : [];
}
```

Appelé dans `todoItemHTML` (render.js), donc une fois par tâche à chaque rendu. `getProjects()` dans `projectManager.js` a un cache (`_projectsCache`) — `getCategories()` devrait avoir la même optimisation.

---

### 🟡 `dayOrder`, `recurringOrder`, `punctualPeriodOrder` — objects potentiellement grands

**Fichier :** `js/app.js` (lignes 81-84)

Ces objets indexés par date s'accumulent indéfiniment dans `localStorage`. Un utilisateur qui utilise l'app depuis 2 ans aurait ~730 clés dans `dayOrder`. Pas de mécanisme de nettoyage des dates anciennes.

### 🟢 `_planMonthGenWeeks` — HTML généré par concaténation dans une boucle

```js

for (let w = 0; w < count; w++) {
  // ...
```

Pour de petits counts (6-20 semaines), ce n'est pas un problème. Pour le scroll infini de la vue plan mois, acceptable.

## 6. Architecture

### 🟡 `window.app` utilisé comme bus de communication global

L'application entière est couplée à `window.app`. Les modules `render.js`, `projectManager.js`, `projectView.js`, `admin.js`, `modal.js` génèrent du HTML avec des `onclick="window.app.xxx()"`. Cela crée un couplage fort et rend les tests unitaires difficiles.

C'est un choix architectural délibéré pour ce type d'app (SPA sans framework), et il est appliqué de façon cohérente. L'alternative (event delegation, pub/sub) serait un refactor majeur. À noter comme dette technique.

---

### 🟡 Duplication de la logique de restauration de backup

La logique de restauration d'un backup (`_syncServer` lignes 200-244 et `_applyBackup` lignes 4481-4522) est substantiellement dupliquée. Les deux blocs restaurent `categories`, `templates`, `suggestedTasks`, `config`, etc. `_syncServer` est plus ancienne et moins complète (elle ne restaure pas `intentions`, `boardProjects`, `avatar`).

Si `_syncServer` (le serveur local) répond, les données `intentions` et `boardProjects` ne seront pas restaurées. `_syncServer` devrait appeler `_applyBackup` pour éviter la duplication et le manque de cohérence.

---

### 🟡 `_applyBackup` — synchronisation config non-atomique

**Fichier :** `js/app.js` (lignes 4503-4512)

La config (theme, zoom, lang, etc.) est toujours appliquée quand le backup Firestore arrive, même si la config locale est plus récente. Il y a bien un `_localConfigTime` dans `_saveConfigChange()` (ligne 254), mais ce timestamp n'est jamais comparé dans `_applyBackup`. La config Firestore peut écraser des préférences locales récentes.

---

### 🟢 `state.js` exporte des variables mutables directement

**Fichier :** `js/modules/state.js`

Les variables d'état (`todos`, `view`, `navDate`, etc.) sont exportées directement et ne sont censées être modifiées que via les setters. Mais JavaScript ES modules ne permet pas d'imposer cela — n'importe quel module importateur pourrait faire `state.todos.push(...)` sans passer par un setter. C'est un pattern fragile mais accepté dans ce contexte. Aucune violation évidente n'a été repérée.

## 8. Points positifs

- **Sécurité iCal token solide** (`api/ical.js`) : lecture directe du document Firestore sans query, comparaison du secret en backend — pas de timing attack possible via requête Firestore.
- **Gestion des échos Firestore** (`sync.js` + `app.js`) : le `SESSION_ID` pour filtrer les mises à jour Firestore émises par la session courante est une solution propre et efficace.
- **Offline-first** : `initializeFirestore` avec `persistentLocalCache()` + file d'attente de writes offline — bien configuré.
- **Undo système robuste** : snapshot avant chaque mutation destructive, cohérent dans tout le codebase.
- **Nettoyage des event listeners** : `_destroyDraftListeners()`, `_destroyCombobox()` — bonne pratique suivie dans modal.js.
- **Cross-tab sync** : `initCrossTabSync` + handler complet dans `app.js` — rare dans les SPA vanilla, bien implémenté.
- **Échappement HTML systématique** : `esc()` utilisé dans tous les endroits clés du rendu (titres, noms de catégories, descriptions).
- **Déduplication des listeners sur `openSettingsMenu`** et `initHeaderDropZones` via des flags — évite les fuites de listeners.
- **Migration des tâches récurrentes sans `startDate`** au démarrage (app.js ligne 93-100) : migration silencieuse et ciblée.
- **iCal serverless** (`api/ical.js`) : génération iCal de qualité, avec EXDATE, RRULE, timezones, filtres — complet et correct.
